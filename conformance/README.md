# DMTAP Conformance

The conformance suite is the **operational definition** of "DMTAP-compatible": an
implementation conforms at a level (Core / Private / Groups&Files / Legacy / Clients / Auth,
see spec ¬ß10) if and only if it passes the corresponding cases here. It also covers the optional,
capability-negotiated **DMTAP-PUB extension** (¬ß22, `pub-1`) and the **CAD/artifact profile**
built on top of it (¬ß23) ‚Äî additive, capability-gated families layered on the Core level exactly
as the optional push-wake (`PUSH`) family is (¬ß10.2, ¬ß21.22).

The suite has two coupled artifacts plus this guide:

| File | Role |
|------|------|
| [`SUITE.md`](SUITE.md) | The **normative test-case catalog**: every case numbered (`DMTAP-<CAT>-<NN>`), grouped by ¬ß10.3 level, with its spec clause, input, expected result (accept / reject + ¬ß21 error code), and MUST/SHOULD. |
| [`suite.json`](suite.json) | The **machine-readable** form of those cases (mirroring their case ids; see the sync note below) ‚Äî a runner in any language drives it. |
| [`vectors/vectors.json`](vectors/vectors.json) | The **byte-exact known-answer inputs** the Core/Private/Groups&Files/Legacy/Clients/Auth vectored cases dispatch on. |
| [`vectors/pub_vectors.json`](vectors/pub_vectors.json) | The **byte-exact known-answer inputs** the `PUB` (¬ß22) vectored cases dispatch on ‚Äî a **separate file**, generated by [`vectors/gen_pub_vectors.py`](vectors/gen_pub_vectors.py), not by `dmtap-core` (see [Provenance & drift protection](#provenance--drift-protection)). |

`SUITE.md` catalogs **157** numbered cases and `suite.json` mirrors all **157** ‚Äî the two are **in
sync** (the wave-2 deniable-1:1 and KT-v1-hardening families, the `PROFILE` display-data cases, the
optional `PUSH` wake-signaling cases, the `FILE` durability cases, the wave-3 device-cluster
`SYNC`, `ALIAS`, and gateway-alias `GWALIAS` families, the wave-4 DMTAP-PUB `PUB` family and the
CAD/artifact profile `CAD` family are all mirrored) and MUST stay so.
**52 cases are byte-runnable today** (46 backed by a `vectors.json` or `pub_vectors.json` entry, 6
self-contained CBOR-reject cases whose bytes are inline); 104 are `construction-todo` ‚Äî each
carries an exact construction recipe and the expected ¬ß21 error, and becomes byte-backed as its
subsystem gains a fixed-input KAT (see [Coverage vs. deferred](#coverage-vs-deferred)); 1
(`DMTAP-PUB-21`) is `manual-attestation` ‚Äî a client-UX MUST with no wire bytes to recompute.

### How a third-party implementer uses the suite

1. Parse `suite.json`. For each case, read `level`, `operation`, `expect.outcome`
   (`match` / `accept` / `reject`), and ‚Äî for `reject` ‚Äî `expect.error_code`/`error_name`.
2. If the case has a `vector` (or `reuses_vector`), load that entry from `vectors.json` ‚Äî or, for a
   `PUB`/`CAD` case, from [`vectors/pub_vectors.json`](vectors/pub_vectors.json) ‚Äî and dispatch on
   its `operation` (see [How an implementation runs the vectors](#how-an-implementation-runs-the-vectors)
   and, for `PUB`, [How an implementation runs the PUB vectors](#how-an-implementation-runs-the-pub-vectors)).
   If it has an inline `input.cbor_hex`, feed those bytes to your deterministic-CBOR decoder.
   If it is `construction-todo`, build the fixture from the `construction` recipe. If it is
   `manual-attestation` (currently only `DMTAP-PUB-21`), there are no wire bytes to recompute ‚Äî
   verify the client-UX requirement by review.
3. Assert the outcome: a `match` case recomputes to the committed value; an `accept` case validates;
   a `reject` case fails **and** maps to the named ¬ß21 code with that code's disposition
   (DROP_SILENT / DEFER_REQUESTS / FAIL_CLOSED_BLOCK / HALT_ALERT / ACK_DEDUP / ROTATE_RETRY /
   DENY_POLICY).
4. You conform at a level iff every `MUST` case at that level (and every level it composes) passes.
   `pub-1` (the `PUB`/`CAD` families) is an **optional capability**, not a level: its guards are
   MUST *only* for an implementation that advertises `pub-1` (¬ß10.2, ¬ß21.22, ¬ß21.24b) ‚Äî the
   identical treatment as the optional `push-wake` (`PUSH`) capability.

### Reference runner

The reference runner is the `dmtap-core` crate's self-checking test
(`crates/dmtap-core/tests/conformance_vectors.rs`, run with `cargo test -p dmtap-core`): it is
exactly the vector-dispatch loop below plus the drift guard of
[Provenance & drift protection](#provenance--drift-protection). It is a **proof**, not the
definition ‚Äî the spec + `SUITE.md` + `suite.json` + `vectors.json` are authoritative (¬ß10.4). A
third-party runner in any language is written directly against those four; it does not need the
reference crate.

## Byte-exact known-answer vectors

[`vectors/vectors.json`](vectors/vectors.json) contains **byte-for-byte known-answer vectors**
for DMTAP's deterministic, security-critical operations. They make the spec
**machine-verifiable**: any independent implementation can compute the same operation over the
same fixed input and check that it gets the same committed answer.

The vectors are **not hand-written** ‚Äî they are computed by the reference crate `dmtap-core`
(`../../envoir/crates/dmtap-core/`) and are proven correct against it by a self-checking test
(see [Provenance & drift protection](#provenance--drift-protection)). Once published, **the
vectors ‚Äî not the reference ‚Äî are normative** (spec ¬ß10): a divergence between the reference and
these vectors is a bug to reconcile.

### Fixed-seed methodology

Every value is derived from a **fixed seed or fixed input** so it is exactly reproducible:

- **Ed25519** is deterministic per **RFC 8032** ‚Äî a given secret seed + message always yields the
  same signature. Keys are reconstructed from fixed 32-byte seeds (`IdentityKey::from_seed`),
  never generated randomly.
- **BLAKE3-256** (content addresses, key-names, safety fingerprints) and **CBOR** (RFC 8949) are
  deterministic functions of their input.
- Two of the sign vectors reuse the **RFC 8032 ¬ß7.1 Test 1 / Test 2** secret seeds, so an
  implementation gets a free cross-check against a published external standard (the committed
  public keys `d75a9801‚Ä¶` / `3d4017c3‚Ä¶` and signatures `e5564300‚Ä¶` / `92a009a9‚Ä¶` are exactly the
  RFC's values).

**Nothing that depends on fresh randomness is vectored as a full object.** HPKE encapsulation
uses an ephemeral key from the OS CSPRNG, so a sealed `Envelope.ciphertext` is not reproducible;
instead the *deterministic sub-operations* it decomposes into ‚Äî the content-address check, the
`sender_sig` preimage/signature, and the `payload_sig` preimage/signature ‚Äî are each vectored as
known-answer tests.

### Vector file format

`vectors.json` is a single object: `{ format, suite, generated_by, methodology, vectors: [...] }`.
Each entry in `vectors` is:

```json
{ "name": "‚Ä¶", "operation": "‚Ä¶", "input": { ‚Ä¶ }, "expected": { ‚Ä¶ }, "note": "‚Ä¶" }
```

`name` is unique. `operation` names the algorithm a runner dispatches on. Byte strings are
lowercase hex. `input` fully determines the operation (for the input-driven operations); `expected`
is what a conforming implementation MUST produce.

## What the vectors prove

| # | Operation(s) | Vectors | What it proves |
|---|--------------|---------|----------------|
| 1 | `content_address` | `content_address_{empty,small,phrase,multi_kib}` | Content address = `0x1e ‚Äñ BLAKE3-256(bytes)` (spec ¬ß2.2, ¬ß18.1.5) for empty, small, phrase, and a 4096-byte input. |
| 2 | `content_address_verify` | `mote_content_address_{ok,tampered}` | The ¬ß2.7 step-2 check: `id` recomputes from `ciphertext`, and a tampered ciphertext is rejected **before any decryption**. |
| 3 | `keyname_encode` / `keyname_verify` | `keyname_{zero_key,key_ones,key_twos,real_pubkey}`, `keyname_typo_rejected` | The zero-authority 8-word + checksum key-name (¬ß3.9.1, ¬ß16.2): determinism, distinctness (`key_ones` ‚âÝ `key_twos`), and that a mistyped word fails the folded checksum (fail-closed). |
| 4 | `safety_number` | `safety_number_pair_ab`, `safety_number_order_independent` | The ¬ß3.4.1 OOB safety number: a deterministic fingerprint of a **pair** of identity keys, **order-independent** (swapping the two keys yields the identical value). |
| 5 | `ed25519_sign` / `ed25519_verify` | `ed25519_rfc8032_test{1,2}`, `ed25519_domain_separated`, `ed25519_verify_{ok,tampered_msg,tampered_sig}` | Deterministic Ed25519 signatures (incl. two RFC 8032 cross-checks and DMTAP's domain-separated `sign_domain`), and that verification fails closed on a tampered message or signature. |
| 6 | `cbor_encode` | `cbor_{identity,device_cert,payload,envelope}` | Exact deterministic CBOR bytes of a signed `Identity` (+ its content address), a `DeviceCert`, a `Payload`, and an `Envelope`. The self-check additionally round-trips each (`decode(encode(x)) == x`, and re-encode is byte-identical). |
| 7 | `suite_decode` | `suite_{reject_0x00,reject_0x04,reject_0x05,reject_0xff,accept_0x01,accept_0x02,accept_0x03}` | Suite fail-closed (¬ß1.1, ¬ß18.1.4): an unregistered suite byte MUST be rejected on decode; `0x01`/`0x02`/`0x03` are known (reserved) ids that decode. |
| 8 | `ed25519_sign` / `ed25519_verify` | `mote_sender_sig`, `mote_sender_sig_verify`, `mote_payload_sig` | The two MOTE signature KATs (¬ß2.7 steps 3 & 8): the `sender_sig` preimage `id ‚Äñ to ‚Äñ ts_be64 ‚Äñ kind ‚Äñ challenge` under the ephemeral key, and the `payload_sig` preimage `BLAKE3-256(CBOR(payload with sig cleared) ‚Äñ u8(kind) ‚Äñ u64be(ts) ‚Äñ CBOR(to))` under the IK ‚Äî both with their exact domain-separation labels. |

The **MOTE validation ordering** (¬ß2.7) is expressed as KATs #2 (content-address, checked before
decryption) and #8 (the two signature checks), which is the deterministic core of the ordered
recipient validation.

## How an implementation runs the vectors

1. Parse `vectors/vectors.json`.
2. For each vector, dispatch on `operation`, feed it the hex/field values in `input`, and compare
   your result to `expected`:
   - `content_address` ‚Üí `0x1e ‚Äñ BLAKE3-256(bytes_hex)` == `id_hex`.
   - `content_address_verify` ‚Üí recompute the address of `ciphertext_hex`; it matches `id_hex` iff `valid`.
   - `keyname_encode` ‚Üí your key-name of `pubkey_hex` == `name` (and it checksum-verifies).
   - `keyname_verify` ‚Üí checksum-verifying `name` == `checksum_verifies`.
   - `safety_number` ‚Üí your safety number / fingerprint of `(ik_a_hex, ik_b_hex)` == `safety_number` / `fingerprint_hex`.
   - `ed25519_sign` ‚Üí `Ed25519(seed_hex).sign(domain_hex ‚Äñ msg_hex)` == `sig_hex`, and the public key == `pubkey_hex`.
   - `ed25519_verify` ‚Üí verifying `sig_hex` over `domain_hex ‚Äñ msg_hex` under `pubkey_hex` == `valid`.
   - `suite_decode` ‚Üí decoding the CBOR in `cbor_hex` as a suite id succeeds iff `accepted`.
   - `cbor_encode` ‚Üí decode `cbor_hex`; re-encoding MUST be byte-identical (deterministic CBOR).

The reference runner is the crate's own self-check test, which is exactly this loop plus a drift
guard (below); use it as the executable specification of the dispatch.

## Provenance & drift protection

The vectors are generated and continuously verified by `dmtap-core`:

- **Generate:** `cargo run -p dmtap-core --example gen_vectors` recomputes `vectors.json` from the
  reference crate.
- **Self-check:** `cargo test -p dmtap-core` (test file `tests/conformance_vectors.rs`) proves,
  on every test run, that
  1. every input-determined vector **re-derives from `dmtap-core`** to its committed `expected`
     value (the vectors are correct against the reference, not guessed), and
  2. the committed `vectors.json` **byte-for-byte matches** what the current reference generates ‚Äî
     so if a primitive ever changes, the test fails until the vectors are regenerated (**drift is
     caught**),

  plus CBOR round-trips and the suite fail-closed property directly.

The generator and the checker share one source of truth (`crates/dmtap-core/vectors_gen.rs.inc`),
so they cannot silently disagree.

## Coverage vs. deferred

**Covered (deterministic, known-answer):** content addressing; the key-name (encode + checksum);
safety numbers; Ed25519 sign/verify (incl. RFC 8032 cross-checks + domain separation); canonical
CBOR of `Identity` / `DeviceCert` / `Payload` / `Envelope` (+ round-trip); suite fail-closed; and
the MOTE content-address and signature KATs. **33 vectors** across 9 operations.

**Deferred, and why:**

- **HPKE seal/open (suite 0x01) and any full sealed `Envelope.ciphertext`** ‚Äî RFC 9180 HPKE uses
  an ephemeral encapsulation key from the CSPRNG, so the output is non-deterministic. `dmtap-core`
  exposes no fixed-randomness HPKE path, so a byte-exact seal vector is not reproducible. Its
  deterministic decomposition (content-address + `sender_sig` + `payload_sig`) is covered instead.
  A fixed-KAT HPKE vector could be added if the reference gains a deterministic-RNG test seam.
- **MLS group operations (`GroupState`, `GroupEvent`, Welcome/Commit, ¬ß5)** ‚Äî MLS handshake bytes
  (RFC 9420) are carried opaquely by DMTAP and are not implemented in `dmtap-core`.
- **Suite `0x02` (PQ: ML-DSA-65 / X-Wing)** ‚Äî reserved and unimplemented; it correctly fails
  closed, which *is* vectored (`suite_reject_*` covers unknown-suite rejection; a positive
  `0x02` object KAT awaits an implementation).
- **Argon2id PoW, ARC tokens, postage, vouches (¬ß9), key-transparency proofs (¬ß3.5), Merkle-DAG
  manifests (¬ß5.5), DMTAP-Auth challenge/response (¬ß13), name‚Üíkey DNS/KT resolution (¬ß3.3)** ‚Äî
  either not yet implemented in the reference or dependent on external/interactive state; to be
  added as those subsystems land.

### A note on the CBOR encoding vectored here

The `cbor_encode` vectors are the **¬ß18-canonical, integer-keyed (COSE/CWT-style) deterministic
encoding** (RFC 8949 ¬ß4.2; spec ¬ß18.1.1 / ¬ß18.1.2 / ¬ß18.3.x / ¬ß18.4.x), matching the
`vectors.json` `methodology` field and each vector's own note. This is directly visible in the
bytes: e.g. `cbor_identity` begins `a9 01 81 01 02 a1 ‚Ä¶` ‚Äî a 9-entry map keyed by the small
unsigned integers `1, 2, 3, ‚Ä¶` in ascending order, exactly as ¬ß18.4.1 specifies, **not** a
text-keyed map. A second implementer following ¬ß18 alone reproduces these bytes; the self-check
additionally round-trips each object (`decode(encode(x)) == x`, byte-identical) and recomputes the
`Identity` content address (¬ß18.9.4). Cases `DMTAP-CBOR-01‚Ä¶04` in `SUITE.md` pin these four
objects; `DMTAP-CBOR-05‚Ä¶12` pin the canonical-form **reject** rules (non-preferred integers,
indefinite lengths, unsorted/duplicate keys, floats, `undefined`, null-valued optionals, and
unknown `‚â• 64` keys in signed objects).

## The DMTAP-PUB (¬ß22) vectors ‚Äî `pub_vectors.json`

[`vectors/pub_vectors.json`](vectors/pub_vectors.json) is the byte-exact known-answer companion to
the `PUB` cases (`DMTAP-PUB-01`‚Ä¶`21`) and, indirectly (no bytes of its own, ¬ß23.1), the `CAD`
checklist (`DMTAP-CAD-01`‚Ä¶`11`). It is a **separate file** from `vectors.json` with its own
`format`/`suite`/`generated_by`/`methodology` header, kept distinct rather than merged into
`vectors.json` because the two have genuinely different provenance (below) ‚Äî merging them under
one `generated_by` would misrepresent where the bytes came from.

### Provenance: a script, not `dmtap-core`

`vectors.json` is generated and drift-checked by the `dmtap-core` reference crate (above).
`pub_vectors.json` is **not** ‚Äî `dmtap-core` does not yet implement the DMTAP-PUB extension
(ROADMAP.md: "Envoir node/gateway `pub-1` serving" is still a follow-up wave). Every operation
DMTAP-PUB needs for a byte-exact vector is nonetheless fully deterministic ‚Äî BLAKE3-256 content
addressing (the same primitive `vectors.json` uses) and Ed25519 signing over a domain-separated
preimage (the same `sign_domain` pattern as `mote_sender_sig`/`ed25519_domain_separated`) ‚Äî so
[`vectors/gen_pub_vectors.py`](vectors/gen_pub_vectors.py), a ~300-line throwaway Python script
with **fixed** Ed25519 seeds (`0xAA`√ó32, `0xBB`√ó32), a **fixed** timestamp
(`1700000050000` ms, in the style of `vectors.json`'s own `mote_payload_sig` vector), and
**fixed** plaintext chunk bytes (no randomness, no wall-clock reads), reproduces every vector
below independently of any reference implementation. Regenerate/verify with:

```
pip install blake3 cryptography
python3 conformance/vectors/gen_pub_vectors.py > conformance/vectors/pub_vectors.json
```

Every value in this file was independently re-derived ‚Äî a second, from-scratch implementation of
the same BLAKE3/Ed25519/CBOR-decode formulas that deliberately does not import
`gen_pub_vectors.py` ‚Äî by [`vectors/verify_pub_vectors.py`](vectors/verify_pub_vectors.py) during
the wave-4 review that added it (`python3 conformance/vectors/verify_pub_vectors.py`; exits
nonzero on any mismatch). A divergence between the generator, the verifier, and the committed file
is a bug to reconcile, the same rule ¬ß10 states for `vectors.json`/`dmtap-core`.

### How an implementation runs the PUB vectors

1. Parse `vectors/pub_vectors.json` (a separate parse from `vectors/vectors.json`).
2. For each vector, dispatch on `operation`:
   - `pub_manifest_root` ‚Üí hash each `plaintext_chunks_hex[i]` to `h_i = 0x1e‚ÄñBLAKE3-256(...)`
     (== `chunk_hashes_hex[i]`), fold them into the ¬ß22.2.2 DS-tagged tree, and check the root ==
     `id_hex`.
   - `pub_manifest_type_mismatch` ‚Üí root the given `chunk_hashes_hex` both the public (DS-tagged)
     and the ¬ß18.9.5 bare-sealed way; assert `public_root_hex` ‚âÝ `sealed_style_root_hex`.
   - `det_cbor_decode_pub_manifest` / `det_cbor_decode_feed_entry` ‚Üí your deterministic-CBOR
     decoder of a `PubManifest`/`FeedEntry` MUST reject `input.cbor_hex`.
   - `ed25519_sign` ‚Üí identical dispatch to `vectors.json`'s `ed25519_sign`:
     `Ed25519(seed_hex).sign(domain_hex ‚Äñ msg_hex)` == `sig_hex`, public key == `pubkey_hex`.
   - `content_address` ‚Üí identical dispatch to `vectors.json`'s `content_address`:
     `0x1e‚ÄñBLAKE3-256(bytes_hex)` == `id_hex`.
   - `pub_supersede_check` / `pub_feed_anti_rollback` ‚Üí validation predicates; compare your
     accept/reject decision (and, on reject, `error_code`/`error_name`/`action`) to `expected`.
   - `pub_feed_entry_root` ‚Üí recompute each `entries_cbor_hex[i]`'s content address; check against
     `entry_ids_hex[i]` and that the `prev` fields resolve correctly along the chain.

There is no `dmtap-core`-equivalent self-checking test for these vectors in this repo (no Rust
reference implements DMTAP-PUB yet); `gen_pub_vectors.py` **is** both the generator and, by
re-running it and diffing, the drift check.

## The substrate Sync capability vectors ‚Äî `sync_vectors.json` (informative companion)

[`../substrate/SYNC.md`](../substrate/SYNC.md) is an **additive substrate companion document**, not a
numbered section (`substrate/README.md`); its own ¬ß10 conformance-vector table is the authoritative
index. [`vectors/sync_vectors.json`](vectors/sync_vectors.json), generated by
[`vectors/gen_sync_vectors.py`](vectors/gen_sync_vectors.py), freezes 15 of its 20 stubs ‚Äî the same
throwaway-script provenance model as `pub_vectors.json` above (no reference implementation exists for
this wire shape; `dmtap-core` does not implement it). The remaining 5 stubs are left as construction
recipes with an explicit NOT-FROZEN reason each (an unfrozen `COSE_Sign1` envelope, an unfrozen
"observable state" snapshot schema, and an unfrozen range-Merkle fold function) ‚Äî see `SYNC.md` ¬ß10 for
the per-stub reasoning; freezing those would mean this repo inventing a design choice the substrate
document itself has not made. This section is listed here only for discoverability; `SYNC.md` ¬ß10, not
this README, is normative for what each vector proves.
